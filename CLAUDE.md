# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm run dev       # start dev server (Vite HMR)
npm run build     # production build
npm run preview   # preview production build locally
npm run lint      # run oxlint (whole project)
npx oxlint src/hooks/usePomodoro.js   # lint a single file
npm test          # run the Vitest suite once
npm run test:watch # run Vitest in watch mode
```

Tests run under Vitest with jsdom (config lives in the `test` block of `vite.config.js`). Hook tests use `@testing-library/react`'s `renderHook` and fake timers to drive the countdown without waiting on real time ‚Äî see `src/hooks/usePomodoro.test.js` for the pattern. Browser-only side effects (audio, notifications) live in `src/lib/alert.js` and are mocked with `vi.mock` in tests, since jsdom doesn't implement `AudioContext`/`Notification`. Pure calculation helpers used by a component (e.g. `countAvailablePomodoros` in `src/lib/pomodoroMath.js`, `isCurrentBlock` in `src/lib/timetable.js`) live in their own `src/lib/*.js` file rather than being exported from the component ‚Äî co-exporting a plain function from a component file breaks Fast Refresh (oxlint's `only-export-components` warns on this).

## Architecture

This is a Pomodoro timer app implementing the Pomodoro Technique (25-min work sessions, short/long breaks, interruption tracking). Stack: React 19, Vite 8, Tailwind CSS v4, oxlint.

### Data flow

All persistent state lives in `localStorage`. `src/lib/storage.js` is the single point of access ‚Äî it exposes typed load/save helpers for its storage keys:

| Key | Purpose |
|-----|---------|
| `pomodoro_inventory` | Activity Inventory ‚Äî the task backlog |
| `pomodoro_today_tasks` | Tasks selected for today with realized/interruption counts |
| `pomodoro_activity_log` | Completed-task records (estimate vs. actual) |
| `pomodoro_ticks` | Lightweight per-pomodoro/interruption events for reports |
| `pomodoro_settings` | User preferences: `cycleLength`, `shortBreakMinutes`, `longBreakMinutes`, `theme`, `chimeStyle` |
| `pomodoro_categories` | User-defined categories (name + color) ‚Äî see Categories below |
| `pomodoro_void_log` | Voided-Pomodoro journal entries (reason, elapsed time) ‚Äî see Void log below |

`patchSettings(partial)` merges into existing settings and must be used instead of `saveSettings` whenever a feature only owns one key of the settings object ‚Äî otherwise it clobbers the others. `exportAllData()` bundles every key for the JSON backup download in `RecordsLog`.

`storage.js` also owns a `pomodoro-data-changed` custom window event. Any write that should notify passive listeners (adding a tick, completing a task) dispatches this event. Components that read from storage but don't receive props ‚Äî `RecordsLog` and `Reports` ‚Äî subscribe with `subscribeToChanges()` to re-read when data changes.

**Backend-readiness prep:** every `loadJSON`/`saveJSON` call goes through an internal `localStorageProvider` object (`get`/`set`/`remove`, keyed by collection) instead of touching `localStorage` directly ‚Äî swapping in a remote backend later means replacing that one object, not every call site. Every normalized record also carries `userId` (always `'local'` for now), `createdAt`, and `updatedAt` ‚Äî old records without them just get `null` timestamps, same `?? null` pattern the rest of normalization already uses. `storage.js`'s own `addActivityRecord`/`addTick`/`addVoidLogEntry` stamp `createdAt`/`updatedAt` at creation (`stampCreated`); `updateActivityRecord` bumps `updatedAt` (`stampUpdated`). Inventory/Today-tasks/Timetable/Categories have no per-record add/update functions in `storage.js` ‚Äî hooks `saveX` the whole array instead ‚Äî so those only gain real timestamps once such granular functions exist; this is schema prep, not a working sync layer (still no async I/O, realtime channel, or auth).

### Hook layer

Five hooks own all mutable state:

- **`useInventory`** ‚Äî CRUD for the Inventory list; auto-saves to localStorage on every change. Also owns `combineItems` (Rule 5: merge 2+ small tasks into one, summing estimates).
- **`useTodayTasks`** ‚Äî manages today's task list: realized pomodoro count, internal/external interruption counters, active task selection, re-estimates (`reestimateTask`, up to `reestimate1`/`reestimate2`). Calling `finishTask` writes a record (with `diff`/`diffI`/`diffII`) to the activity log and marks the task done.
- **`usePomodoro`** ‚Äî pure timer state machine (work ‚Üí shortBreak/longBreak cycle). After every `cycleLength` completed pomodoros (user-configurable, default 4) it automatically switches to a long break. It writes a tick on each completed work session and interruption. Work/short-break/long-break lengths are all user-configurable (`workMinutes`/`shortBreakMinutes`/`longBreakMinutes`, defaults 25/5/15) via `setWorkMinutes`/`setShortBreakMinutes`/`setLongBreakMinutes`, matching the same `patchSettings`-on-change pattern as `setCycleLength`. Short/long break clamp to hard bounds (3‚Äì10 and 15‚Äì60, exported as `SHORT_BREAK_MIN/MAX`/`LONG_BREAK_MIN/MAX`) with a non-blocking `SettingsTab.jsx` hint when the value exceeds methodology.md's recommended max (5 / 30) ‚Äî still fully accepted past that point. Work duration has no recommended-range concept at all (per methodology, 25 minutes just *is* the technique) ‚Äî only the sane absolute bounds `WORK_MIN`/`WORK_MAX` (1‚Äì180) apply, and `SettingsTab.jsx` instead shows an informational note whenever it differs from `DEFAULT_WORK_MINUTES` (25), framed as a deliberate deviation rather than a recommendation to revert. An idle (not-running) session's displayed countdown stays in sync with its type's current setting via a dedicated effect ‚Äî so changing a duration takes effect the next time Start is pressed, not just after one extra full cycle ‚Äî but a session that's actually running is never resized out from under the user (the effect no-ops while `isRunning`). `Timer.jsx` mirrors these three values itself (via the same `pomodoro` prop spread) to compute the ring's progress arc, since it has no other way to know each session type's full duration.
- **Auto-start** ‚Äî two independent, default-off toggles: `autoStartBreaks` (a completed Pomodoro's break begins immediately instead of waiting for Start) and `autoStartPomodoros` (a finished break's next work session begins immediately). Both are consulted at the end of `completeWork`/`completeBreak` respectively; `skipBreak` also honors `autoStartPomodoros` (skipping is still "a break ending," just early) ‚Äî manual `switchSession` (the session-switch pills) deliberately does not auto-start either way, since that's an explicit user choice of where to jump, not a natural transition.
- **Sound volume** ‚Äî `soundVolume` (0‚Äì100, default 100) is applied globally via `alert.js`'s own module-level `setVolume(percent)`/`volume` (0‚Äì1), which every `playTone` call scales its peak gain by ‚Äî so `usePomodoro` only needs a `useEffect` calling `setAlertVolume(soundVolume)` on change, rather than threading a volume argument through every `playChime`/`playPing`/`playTaskCompleteChime` call site. `alert.js` floors the scaled peak at `0.0001` rather than letting it reach exactly `0`, since Web Audio's `exponentialRampToValueAtTime` throws on a zero target ‚Äî reachable for real at the volume slider's minimum.
- **Ambient background sound** ‚Äî `ambientSound` (`alert.js`'s `AMBIENT_SOUNDS`: `'none'` default, `'ticking'`, `'rain'`, `'cafe'`, `'whiteNoise'`) plays for as long as `ambientSound !== 'none' && isRunning && sessionType === 'work'` all hold, via one `useEffect` keyed on exactly those three values ‚Äî pause/void/completion/break all naturally stop it since each flips at least one of them. `alert.js`'s `startAmbientSound(kind)`/`stopAmbientSound()` route to one of two implementations: `'ticking'` keeps the original discrete-click approach (a `setInterval` re-triggering a short `playTone` once a second ‚Äî its volume is read fresh every tick, no live-update needed); `'rain'`/`'cafe'`/`'whiteNoise'` share a single noise-buffer audio graph (a looping `AudioBufferSourceNode` filled with random samples ‚Üí an optional `BiquadFilterNode` ‚Üí a `GainNode` ‚Üí destination), differing only in which filter (if any) shapes the same raw noise ‚Äî unfiltered for white noise, band-passed for rain, low-passed (plus occasional random high-pitched "clink" blips) for cafe. All are simple Web Audio approximations, not recordings, consistent with this file's no-external-assets approach. Since these beds are *continuous* (unlike the one-shot tones), `setVolume()` also live-updates the currently-playing bed's gain node, not just the module-level `volume` multiplier the one-shot sounds read fresh each time. A legacy boolean `tickingSoundEnabled` setting migrates automatically (`storage.js`'s `normalizeAmbientSound`): `true` ‚Üí `'ticking'`, `false` ‚Üí `'none'`, only when no `ambientSound` value has been saved yet.
- **`useTimetable`** ‚Äî today's planned time blocks (`Timetable.jsx`); blocks are stamped with the date they were created and pruned on load if they're from a previous day.
- **`useCategories`** ‚Äî CRUD for user-defined categories (name + color, from `CATEGORY_COLORS` in `constants.js`). No cascade delete: `removeCategory` just drops the category, and anything referencing its id afterward fails to resolve it and falls back to "no category"/"Uncategorized" ‚Äî the same fallback legacy/free-text `type` data already needed, so it's free.

### Component wiring

`App.jsx` instantiates `useInventory`, `useTodayTasks`, `usePomodoro`, and `useCategories`, then passes their data down as props ‚Äî `Timer` and `SettingsTab` are both purely presentational and share the one `usePomodoro` instance (`SettingsTab` needs `cycleLength`/`chimeStyle` setters that live in the same hook Timer renders from, so it can't be timer-local anymore). `useTimetable` is instantiated inside `TodoToday` (not `App`) so its block totals can feed `AvailablePomodoros`' suggested-hours button. The hooks never talk to each other directly ‚Äî `App`/`TodoToday` bridge them: when a task from Inventory is sent to Today it stores the source `inventoryId`, `categoryIds`, and `notes` on the today-task, and when that task is finished `App` also removes it from Inventory. Marking a task done is deliberately decoupled from the timer ‚Äî finishing a task early doesn't stop a running Pomodoro (the remaining time is still there for overlearning). A "Check to bottom" setting (`checkToBottom`, default off, managed in `App.jsx` alongside `theme`/`onboardingDismissed` since it's not owned by any one hook) additionally calls `useTodayTasks`' `moveTaskToEnd(id)` right after `finishTask` when enabled ‚Äî `useTodayTasks` itself stays unaware of Settings, same separation as everywhere else in this bridging layer. Moving to the end of the *array* lands a task at the end of whichever of planned/Unplanned-&-Urgent `TodoToday.jsx` filters it into (filtering preserves relative order), which reads as "bottom of its own section," not "bottom of everything."

`RecordsLog` reads storage directly on mount and on `pomodoro-data-changed` events, but takes `categories` as a prop (from `categoriesApi.categories`) rather than loading it itself ‚Äî it's read-only reference data needed for display/filtering, same reasoning as `Reports`' `todayTasks` prop below. `Reports` does the same for `ticks`/`activityLog`, but also takes two props ‚Äî `todayTasks={todayApi.tasks}` (for its "active task" count: `todayTasks` isn't date-stamped and `saveTodayTasks` doesn't dispatch `pomodoro-data-changed`, so reading it from storage on that event would show stale data) and `categories` (for the category-breakdown chart). `Reports` also renders `DayReview` (an end-of-day summary modal, unchanged by the metrics redesign below).

### Inventory row controls

`Inventory.jsx`'s `InventoryRow` has three independent, easy-to-conflate controls: the checkbox is Rule 5 combine-selection (`selected`, local `Set` state in `Inventory`, not persisted ‚Äî purely a UI selection, unrelated to `done`), the circular button is the actual `done` toggle (`toggleDone`, persisted, and the only thing the strikethrough style is bound to), and the "U" badge is `item.unplanned`. A selected row gets a `bg-tomato/10`/`border-tomato/30` tint (same token pairing as the active-row highlight in `TodoToday.jsx`/`Timetable.jsx`) so combine-selection has visible feedback beyond the "Combine" banner appearing. `unplanned` is set once at add-time but is also editable in edit mode (a "U" toggle button identical to the add-form's, persisted via `updateItem`) ‚Äî unlike categories/deadline/notes, it previously had no way to be changed after creation.

### Categories

Replaces what used to be a free-text `type` field on Inventory/Today-task/Records with a `categoryIds` array (multi-select tags, not a single pick) into `useCategories`' list. Assignment uses `CategoryTagPicker.jsx` (checklist-style multi-select dropdown); `CategorySelect.jsx` is a separate, single-pick dropdown kept only for Records Log's category *filter* (filtering by one category at a time is still the simpler, appropriate UX there ‚Äî it was never converted to multi-select). Both are from-scratch dropdowns like `Select.jsx` but with a leading color-dot per option, built as their own components so the existing sound-style `Select.jsx` isn't put at risk. `CategoryManager.jsx` (rendered in the Settings tab) is the create/edit/delete UI, using a curated 8-color palette (`CATEGORY_COLORS` in `constants.js`) rather than reusing `tomato`/`amber`, which already carry other meanings elsewhere (work/danger, break/"took less") that a category swatch in the same color would collide with. A task's pomodoros count fully toward EACH of its assigned categories in the Reports breakdown ‚Äî not split ‚Äî see `pomodorosByCategory()` in `reportsMath.js`.

Migration: `storage.js`'s `normalizeCategoryIds()` wraps a legacy single `categoryId` (string or `null`) into a one-element array (or `[]`), so old single-select data loads without any explicit migration step ‚Äî normalize functions rebuild an explicit object rather than spreading, so the old key is simply never read again.

Every task/record also gained a `notes` field (long-text description) ‚Äî Inventory already had one pre-existing (upgraded from a single-line input to a `<textarea>`); Today's Tasks and Activity Log records are new. Displayed collapsed behind a üìù toggle in list views (`Inventory.jsx`, `TodoToday.jsx`'s `TaskRow`, `RecordsLog.jsx`'s `RecordRow`) so it doesn't clutter the row ‚Äî expands on click.

The `pairWith` field (a lightweight "pairing with" note, no real-time sync) has been removed entirely ‚Äî categories now cover that use case. Old records/tasks with a legacy `type` or `pairWith` value load fine; `storage.js`'s normalize functions simply don't read those keys anymore, so they're silently dropped rather than migrated.

### Fullscreen Focus Mode

`Timer.jsx` renders one shared JSX tree for both the normal card view and the fullscreen minimal view, toggling extra chrome on/off with `{!isFullscreen && (...)}` guards (session-switch pills, the "Finish Pomodoro" button, the "?" keyboard-shortcuts button, `UnplannedCapture`) rather than maintaining two separate component trees. `isFullscreen` is driven only by the native `fullscreenchange` event (not set optimistically on click) ‚Äî `requestFullscreen()` requires a user gesture and can still be rejected in some contexts, so relying on the event keeps the UI honest about actual fullscreen state regardless of how it was entered/exited (button, `F`, or the browser's own Escape-to-exit).

Keyboard shortcuts: `F` toggles fullscreen (was previously bound to Finish Pomodoro); `E` is now Finish Pomodoro; `Escape` closes the Keyboard Shortcuts modal if it's open, else opens the Void-reason prompt (see below) when a Pomodoro is running, cancels that prompt if it's already open, and does nothing at all while fullscreen (the browser's own Escape-exits-fullscreen handling takes over ‚Äî the handler explicitly no-ops there so it doesn't *also* open the void prompt underneath). `?` opens the Keyboard Shortcuts modal (see below). `Space` is unchanged. There is deliberately no keyboard shortcut for manually switching Focus/Short Break/Long Break (removed `1`/`2`/`3`) ‚Äî the session-switch pills are still click-only.

### Void log

Voiding a Pomodoro (Rule 1) opens an inline panel (`Timer.jsx`, replacing the old bare `window.confirm`) with an optional free-text reason, combining confirmation and reason-capture into one step. `usePomodoro.js`'s `voidPomodoro(reason)` computes elapsed time itself (`DURATIONS.work - secondsLeft`) and reports `{ reason, elapsedSeconds }` via a new `onVoid` callback ‚Äî mirroring the existing `onWorkComplete`/`onInterruption` bridge pattern ‚Äî since the hook has no knowledge of the active task; `App.jsx`'s `onVoid` implementation looks up the active task/its categories and writes the entry via `addVoidLogEntry()`.

### Picture-in-Picture mini timer

`Timer.jsx` uses the Document Picture-in-Picture API (`window.documentPictureInPicture`, feature-detected via `isPipSupported()` in `src/lib/pip.js` ‚Äî the toggle button simply doesn't render when unsupported, e.g. Safari; `document.title`'s existing countdown remains the universal fallback). The PiP window's content is a `ReactDOM.createPortal` into `pipWindow.document.body`, so it re-renders in lockstep with the main countdown automatically ‚Äî no second timer, nothing to keep in sync manually. `copyStylesToWindow()` (also in `pip.js`) copies every stylesheet from the main document into the new (otherwise blank) PiP document, the pattern the API's docs recommend, so the portal content picks up the app's existing Tailwind classes/color tokens; the portal's own root div also re-applies the `.light` theme class (Timer now receives `theme` as a prop from `App.jsx` for this, since it has no other reason to know about theme). `pipWindow` state (open Window object, or `null`) is driven by the PiP window's own `pagehide` event rather than set optimistically, so it can't drift out of sync whether the user closes it from the mini window, the toggle button, or by switching away ‚Äî same reasoning as `isFullscreen` above. Deliberately read-only content (no interruption buttons, no controls) ‚Äî it exists purely so the countdown stays visible when the user is in another tab/app, not as a second control surface.

This is deliberately a simple daily journal, not a metric: `Reports.jsx`/`reportsMath.js` never read `pomodoro_void_log` at all. `RecordsLog.jsx` shows entries in an unobtrusive "Voided Pomodoros" section (own `VoidLogRow`, capped at 5 most recent) that is **not** affected by the Records Log date/category filters ‚Äî those filters exist for the Activity Log's aggregation-adjacent use, not this.

### Reports metrics

`Reports.jsx` is organized around `src/lib/reportsMath.js` (pure, unit-tested date/aggregation helpers ‚Äî `effectiveDiff`, `datesForPeriod`/`datesForThisWeek`/`datesForLastWeek`/etc., `estimationBreakdown`, `avgInterruptionsPerTask`, `trendDirection`). Per methodology, metrics favor trends and per-task ratios over cumulative totals:

- **Today** ‚Äî fixed today-vs-yesterday comparison (pomodoro count, task count, interruptions), independent of the time filter below ‚Äî except for its "Total focus time" card, which is the one deliberate exception: `reportsMath.js`'s `totalFocusMinutes(ticks, dates, workMinutes)` multiplies the *selected period's* completed-Pomodoro tick count by the current `workMinutes` setting (passed down from `usePomodoro` via `App.jsx` as a prop), formatted H:MM by `formatFocusDuration()`. Ticks record only that a Pomodoro completed, not how long it actually ran, so this is an approximation ‚Äî "N pomodoros at today's current work length" ‚Äî the same assumption `AvailablePomodoros`' capacity estimate already makes elsewhere, not a literal historical total that accounts for `workMinutes` having changed over time.
- **Estimation Accuracy** ‚Äî overestimated/underestimated counts and a per-task diff chart, both scoped to the top time filter (Today/Week/Month/Year, default Week); plus a fixed this-week-vs-last-week avg-error trend.
- **Interruption Trends** ‚Äî average interruptions *per finished task* (not a raw total ‚Äî the deliberate correction over the old design), scoped to the time filter, plus a fixed week-over-week trend and a per-task breakdown list.
- **Pomodoros by Category** ‚Äî `pomodorosByCategory()` sums each finished record's `real` grouped by `categoryId`, scoped to the time filter, rendered as a horizontal bar per category colored with that category's own color (an "Uncategorized" bucket, sage-colored, catches null/deleted-category records). This only counts pomodoros from *finished* tasks ‚Äî ticks (the raw pomodoro events) don't carry a category, so there's no way to attribute an in-progress task's pomodoros to a category without adding new tracking.
- **Long-term** ‚Äî collapsed by default; the 13-week heatmap plus a month/quarter summary.

`effectiveDiff` also lives here now (moved out of duplicate copies previously in `Reports.jsx` and `DayReview.jsx`); `DayReview.jsx` imports it from `reportsMath.js`.

**Empty states** ‚Äî three distinct "nothing to show" messages, not to be confused with each other:
- **No data anywhere** (`ticks.length === 0 && activityLog.length === 0`) ‚Äî replaces the entire period-filter row and all sections with a single calm message ("Complete a few Pomodoros to see your trends here."); there's nothing meaningful to filter yet.
- **`hasNoHistoryYet()`** (existing, unchanged) ‚Äî there *is* data, but every tick/record is dated today, so Today/Week/Month/Year would all show the identical result. An italic hint above the sections, not a replacement.
- **Per-period empty** (new) ‚Äî the selected period specifically has zero matching records (most commonly: "Today" selected on a day nothing's finished yet), even though other periods have data. `EstimationAccuracySection`/`InterruptionTrendsSection`/`CategoryBreakdownSection` each independently check their own `recordsInDates(activityLog, datesForPeriod(period))` and substitute a shared `t('reports.noDataForPeriod')` message for their entire content when empty, rather than rendering stat boxes full of zeros next to an empty chart (which is what looked broken before this fix). `EstimationAccuracySection` didn't have any such check previously ‚Äî `InterruptionTrendsSection` and `CategoryBreakdownSection` already had the equivalent logic under differently-worded keys, now unified onto the shared key.

### Header

`App.jsx`'s header (top-right, next to the app title on the left) holds, in order: the date/time, a gear icon button (`setActiveTab('settings')` ‚Äî turns tomato when Settings is the active tab), and `ProfileMenu.jsx` (see Authentication below). None of these are part of `TabNav` ‚Äî Settings is reachable only via the gear icon now, not as a fourth tab (see Tab layout below).

### Keyboard Shortcuts modal

`KeyboardShortcutsModal.jsx` (same `DayReview.jsx`-style modal shell) lists every Timer-tab shortcut in a two-column Command/Action table, driven by a small `SHORTCUTS` array so adding one later is a one-line change. Opened via a small "?" icon button next to Fullscreen/PiP (hidden in fullscreen, same gate as PiP) or by pressing `?` itself ‚Äî both handled in `Timer.jsx`'s existing keydown listener, which also closes it on `Escape` (checked before the void-prompt's own Escape handling, so pressing Escape with the modal open never also opens the void prompt underneath). This replaced a small inline hint line that used to sit under the timer controls.

### Tab layout

`App.jsx` renders three tabs via `TabNav` (`activeTab` state: `timer` | `planning` | `reports` | `settings` ‚Äî `settings` is a valid value but reachable only via the header's gear icon, not through `TabNav` itself) to keep the Timer screen distraction-free per methodology ‚Äî no Inventory/Records/Reports/Settings content should be visible while a Pomodoro is running. **All four tab panels stay mounted; only the active one is toggled visible with Tailwind's `hidden` class** (not conditionally rendered). This is required, not just an optimization: unmounting the Timer panel on tab switch would stop `usePomodoro`'s countdown interval while the user is on another tab, effectively pausing a running Pomodoro. Keeping `RecordsLog`/`Reports` mounted also avoids dropping and re-subscribing their `pomodoro-data-changed` listeners on every switch.

- **Timer tab** ‚Äî `Timer.jsx`: ring + state label, current active task, Start/Void/Finish-early/Skip-break, interruption buttons (internal/external, gated on `isRunning`), and `UnplannedCapture` (add-only input, no list ‚Äî jot it down and keep working).
- **Planning tab** ‚Äî `Inventory` + `TodoToday` (which itself includes `AvailablePomodoros`, `Timetable`, the today-task list, and the full Unplanned & Urgent box with its list). `TodoToday.jsx`'s header row also has a "..." `TaskListMenu` (same click-outside dropdown pattern as `Select.jsx`/`ProfileMenu.jsx`) with two bulk actions ‚Äî "Clear finished tasks" (`useTodayTasks`' `clearFinishedTasks`, disabled when nothing's done) and "Clear all tasks" (`clearAllTasks`, disabled when the list is empty) ‚Äî each still gated behind its own `window.confirm`. Both only ever touch the array in memory/storage; neither is timer-aware.
- **Reports tab** ‚Äî `Reports` (which renders `DayReview` internally) + `RecordsLog` (which also has its own date-picker + category filter, combined with AND logic; unfiltered view caps at the 8 most recent records, filtered views show every match). No longer has export buttons ‚Äî see Data import & export below.
- **Settings tab** ‚Äî `SettingsTab.jsx`: cycle length, work/short/long break duration, auto-start toggles, chime/sound + volume + ambient-sound picker, theme toggle, "Check to bottom", and language (see Internationalization below). Signed-in only: "Change Password" (see Authentication below), rendered as its own row right under the title. Also renders `CategoryManager.jsx` (create/edit/delete categories), `DataTransfer.jsx` (see Data import & export below), and a "Danger Zone" card: seven category-scoped delete buttons (Records, Ticks, Today's Tasks + Timetable, Inventory, Timer state, Categories, Void log), a "Reset to Factory Settings" button that additionally wipes `pomodoro_settings`, and (signed-in only) "Delete Account" (see Authentication below). Each button's `window.confirm` names exactly what it deletes; on confirm the corresponding `clear*`/`resetAllData` function in `storage.js` calls `activeProvider.remove(...)` (localStorage or Supabase, whichever is currently active ‚Äî see Cloud sync above), then the page does a hard `window.location.reload()` ‚Äî required because `useInventory`/`useTodayTasks`/`usePomodoro`/`useCategories` only read storage once at mount, so clearing the key alone wouldn't update the already-running UI. The category-scoped clears never touch `pomodoro_settings`; only `resetAllData()` does.

`UnplannedCapture.jsx` is a shared add-only form (calls `addTask(text, null, { unplanned: true, urgent: true })`) used both standalone on the Timer tab and inside `TodoToday`'s Unplanned & Urgent box (which additionally renders the list).

The Timer keeps a "Finish Pomodoro" (finish-early) control even though methodology.md's Rule 2 says a Pomodoro should always ring ‚Äî this is an intentional, documented deviation (see the comment in `Timer.jsx`): the confirm dialog teaches the overlearning rule but leaves the choice to the user, and confirming still records the Pomodoro as complete.

### Data import & export

`DataTransfer.jsx` (Settings tab, titled "Import & Export Data") holds all four data-transfer options in one place ‚Äî JSON Export, CSV Export, JSON Import, CSV Import, each with a one-line description of what it does/doesn't bring back. Export used to live as CSV/JSON buttons in `RecordsLog.jsx`'s header; moved here (this component was previously `DataImport.jsx`, import-only) so a user managing their data isn't split across two tabs. Export itself is unchanged ‚Äî `exportAllData()`/`activityLogToCSV()` + `downloadFile()`, same as before, just called from here.

Import is backed by pure logic in `src/lib/importData.js` and orchestration in `storage.js`'s `importBackup(data, mode)` / `importActivityLogCSV(rows, categories, mode)`. Both file types share the same flow: pick a file ‚Üí parse + validate the structure (`validateBackupShape`/`validateActivityCSV` ‚Äî reject with an inline error and touch nothing on a bad file) ‚Üí an inline "Replace all data" / "Merge with existing" / "Cancel" choice appears ‚Üí `window.confirm` (same severity/style as Danger Zone) ‚Üí write ‚Üí `window.location.reload()` (same reason as the Danger Zone reload above).

- **JSON (full backup)** ‚Äî `validateBackupShape` checks the parsed object's known array fields (`inventory`, `todayTasks`, `activityLog`, `ticks`, `timetable`, `categories`, `voidLog`) are actually arrays, and `settings` (if present) is a plain object. **Replace** writes every included collection (and `settings`) as-is, wholesale. **Merge** resolves each array collection by `id` via `mergeCollectionById` (newer `updatedAt` wins; a record with no `updatedAt` counts as older than one that has a real timestamp; if neither side has one, the existing record wins rather than flipping a coin) ‚Äî and deliberately leaves `settings` untouched even in merge mode, since reconciling data isn't the same thing as silently overwriting the user's current theme/language/cycle-length preferences with whatever the file happened to have at export time.
- **CSV (Records/Activity Log only)** ‚Äî `export.js`'s `CSV_COLUMNS` const is exported so `validateActivityCSV` can check an imported file's header row matches it exactly (same columns, same order) rather than accepting "any CSV." `parseCSV` is a small hand-rolled parser (quoted fields, doubled `""`, embedded newlines in `notes`) mirroring `escapeCSVField`'s output format. CSV rows carry no `id` (see `CSV_COLUMNS`), so merge can't use the JSON path's id+`updatedAt` logic ‚Äî `mergeActivityRecordsByNaturalKey` instead matches on the `(date, time, activity)` triple: a match means "skip, keep the existing record" (no reliable way to tell which is newer without an id or a CSV-side `updatedAt`), a non-match gets appended as a fresh record. `csvRowsToActivityRecords` resolves the CSV's "; "-joined category *names* back to ids via a lookup against the current `categories` list ‚Äî a name that no longer resolves (renamed/deleted/typo) is silently dropped from that record's tags, the same graceful-degradation pattern used everywhere else a category reference fails to resolve.

### Empty states & onboarding

Per-screen empty-state copy (Inventory, Today's Tasks, Records Log ‚Äî see their `emptyState`/`noRecordsEmpty` translation keys) is deliberately calm and action-oriented rather than a blank list or a generic "no data" ‚Äî see Reports metrics above for its three-tier empty-state handling specifically.

**First-launch welcome card** (Timer tab only) ‚Äî shown when `pomodoro_settings.onboardingDismissed` is `false` *and* every collection App.jsx can see is empty (`inventoryApi.items`, `todayApi.tasks`, `categoriesApi.categories`, plus a direct `loadActivityLog()`/`loadTicks()` check for completeness). Dismissing it (`Timer.jsx`'s `onDismissWelcome` prop, wired from `App.jsx`) sets `onboardingDismissed: true` via `patchSettings` ‚Äî permanent, so it won't resurface even if the user later empties their data again via the Danger Zone. Timer's "no active task" state also gained a `t('timer.goToPlanningButton')` link (`onGoToPlanning` prop ‚Üí `setActiveTab('planning')`) ‚Äî both are hidden in Fullscreen Focus Mode, consistent with that mode showing only the methodology-clean core controls.

Tailwind CSS v4 loaded via `@tailwindcss/vite` plugin. Custom design tokens (colors like `bg-pine`, `text-tomato`, `text-cream`, `text-sage`, `text-ink`, `text-amber`) are defined in `src/index.css` as CSS custom properties and referenced throughout components.

### Completion feedback

Small, calm animation + sound pairs mark two moments ‚Äî deliberately not gamified (no confetti/scores):

- **Pomodoro completion** (`completeWork` in `usePomodoro.js`) ‚Äî a `completionPulseKey` counter increments, which `Timer.jsx` turns into a one-shot ~500ms `animate-ring-pulse` class on the ring `<svg>` (subtle scale + `drop-shadow(currentColor)`, so it automatically matches the tomato/amber `accentClass` already applied). Paired with a new `playPing()` (`alert.js`) alongside the existing configurable `playChime(chimeStyle)`. Per methodology a "Pomodoro" is specifically the work session, so both the pulse and the ping fire only on `completeWork`, never `completeBreak` ‚Äî ending a break keeps its original chime-only behavior.
- **Task completion** (`finishTask` in `useTodayTasks.js`) ‚Äî calls a new `playTaskCompleteChime()` (a longer, still-gentle ascending arpeggio). Visually, `TodoToday.jsx`'s `TaskRow` swaps the instant `line-through` for an absolutely-positioned `.task-strike` bar that draws left-to-right once (`task.done` flipping true is a one-time mount, so the CSS animation doesn't replay on later re-renders), and reuses the empty grid-slot placeholder for a `.animate-task-check` checkmark that pops in and fades out.

All three keyframes live in `index.css`, gated behind `@media (prefers-reduced-motion: no-preference)`. The base (non-media-query) rules are the reduced-motion fallback: `.task-strike` defaults to `width: 100%` (a done task should still visibly read as done ‚Äî that's information, not decoration, so it doesn't disappear), while `.animate-ring-pulse`/`.animate-task-check` default to inert/invisible (matching the pre-animation appearance). Sound is untouched by the media query either way ‚Äî nothing in the JS checks `prefers-reduced-motion`, so audio plays the same regardless of motion preference.

### Internationalization (i18n)

Supports English and Turkish. Deliberately hand-rolled rather than a library (e.g. i18next): two locales, no lazy-loading needs, and no plural rules beyond one spot (the Reports heatmap tooltip, handled with explicit `heatmapTooltipOne`/`heatmapTooltipOther` keys rather than a general plural engine) ‚Äî a library would add a dependency and an abstraction layer for features this app doesn't use.

- **`src/lib/i18n/en.js` / `tr.js`** ‚Äî flat-per-namespace nested dictionaries (`timer.start`, `inventory.deleteConfirm`, etc.), kept in sync by hand since both are small.
- **`src/lib/i18n/index.js`** ‚Äî `detectBrowserLanguage()` (Turkish if `navigator.language` starts with `tr`, else English), `resolveLanguage(settingsLanguage)` (uses the explicit setting if present, otherwise auto-detects ‚Äî so auto-detection keeps re-running every load until the user picks a language), `translate(language, key, vars)` (dot-path lookup + `{{var}}` interpolation, falling back to English then to the raw key if missing), and `formatDateLocalized(isoDate, localeTag)` for locale-aware date display (parses `'YYYY-MM-DD'` manually into `new Date(y, m-1, d)` rather than `new Date(isoString)`, which parses as UTC and can roll back a day in negative-offset timezones).
- **`src/lib/i18n/LanguageContext.jsx` + `src/hooks/useTranslation.js`** ‚Äî a React Context is the one deliberate exception to this app's usual prop-drilling-from-`App.jsx` pattern (see Component wiring above): translation strings are needed by nearly every leaf component, so drilling `t` through every intermediate component would be pure boilerplate, unlike theme/categories which only a handful of top-level components need. `LanguageProvider` wraps `<App />` in `main.jsx`; any component calls `useTranslation()` directly for `{ t, language, setLanguage, localeTag }`. Split into two files (context object / provider component / hook) purely to satisfy oxlint's `only-export-components` ‚Äî same reasoning as `pomodoroMath.js`/`timetable.js` being split out from their components.
- **Settings persistence**: `pomodoro_settings.language` defaults to `null` ("not yet explicitly chosen" ‚Äî `resolveLanguage` auto-detects in this case); picking a language in Settings' now-functional Language row (previously a "Coming soon" placeholder) calls `setLanguage`, which persists via `patchSettings` and immediately re-renders every consumer through the Context.
- **`usePomodoro`** is the one exception that does *not* use the Context ‚Äî as a hook (not a component) instantiated once in `App.jsx`, it takes `t` as an options parameter (same bridge-callback pattern as `onWorkComplete`/`onInterruption`/`onVoid`) for its browser-notification strings (`notify(...)` calls in `completeWork`/`completeBreak`). It defaults to a plain `translate('en', ...)` call when `t` isn't passed, so it stays usable standalone (e.g. in tests that don't wrap it in `LanguageProvider`).
- **`reportsMath.js`'s `pomodorosByCategory`** takes an optional `uncategorizedLabel` parameter (default `'Uncategorized'`, for backward-compatible tests) rather than hardcoding the bucket name, since it's a pure lib function with no access to `t()` ‚Äî `Reports.jsx` passes `t('reports.uncategorized')` explicitly.
- **`CATEGORY_COLORS`** (`constants.js`) gained a `key` field per color (`teal`, `plum`, etc.) ‚Äî `CategoryManager.jsx` looks up `t(\`categoryColors.${c.key}\`)` for the swatch tooltip/aria-label instead of the English `name` field, which remains as a dev-facing fallback label.
- **`common.appTitle`** is the one deliberately *un*-translated string in either dictionary ‚Äî both `en.js` and `tr.js` hold the identical literal `'Pomodoro Technique'`, treated as a fixed brand/technique name (per methodology.md's trademark-avoidance note) rather than UI copy. Don't "fix" the Turkish value to a translated form; the duplication across both files is intentional.

### Authentication (Supabase)

Accounts are optional ‚Äî this is additive on top of the existing localStorage-only guest mode, never a replacement for it. `storage.js` itself is untouched by any of this; auth is session state only so far. `supabase/schema.sql` has the full Postgres schema + RLS policies for the eventual data migration.

- **`src/lib/supabaseClient.js`** ‚Äî reads `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` from env (`.env`, gitignored; `.env.example` is the tracked template). Exports `supabase` as `null` when either var is missing, rather than throwing ‚Äî so a checkout without a configured `.env` still runs in guest mode instead of crashing at import time.
- **`src/lib/auth/AuthContext.jsx` + `src/hooks/useAuth.js`** ‚Äî a second Context, structured identically to `LanguageContext`/`useTranslation` (same context-object/provider/hook file split, same reasoning: auth state is needed from deeply nested places like the header's account menu without prop-drilling). `AuthProvider` wraps `<App />` in `main.jsx`, outside `LanguageProvider` (no dependency either way). Exposes `{ user, loading, signInWithGoogle, signInWithEmail, signUpWithEmail, signOut, deleteAccount }`; `user` is `null` for guests and `loading` is only ever true for the brief window before the initial `getSession()` call resolves. All five auth actions no-op with a `NOT_CONFIGURED_ERROR` when `supabase` is `null`.
- **Session sync**: `supabase.auth.onAuthStateChange` (not just the one-time `getSession()` call) keeps `user` current ‚Äî this is also what picks up the session Supabase's client establishes automatically after an OAuth redirect lands back on the app (`detectSessionInUrl` is on by default), so there's no separate "handle the callback" route/page to build.
- **OAuth redirect URL**: built at call time as `window.location.origin + import.meta.env.BASE_URL` (`AuthContext.jsx`'s `signInWithGoogle`) rather than hardcoded, so it resolves to `http://localhost:5173/` in dev and `https://<user>.github.io/pomodoro-app/` on GitHub Pages automatically (`BASE_URL` reflects `vite.config.js`'s `base`, see Gotchas below). **Both exact URLs must be added to the Supabase project's Auth ‚Üí URL Configuration (Site URL / Additional Redirect URLs) in the dashboard** ‚Äî that allow-list is project configuration, not something any client-side code can set.
- **`AuthModal.jsx`** ‚Äî modeled on `DayReview.jsx`'s modal shell (fixed backdrop, centered card, focus moved in on open/restored on close) for visual consistency with the app's other modals. Google button + email/password form (toggling between sign-in/sign-up) + a "Continue without an account" button that just calls `onClose` ‚Äî dismissing it is a no-op with respect to app state, since guest mode was already the running mode. The email/password form has a "Confirm password" field in sign-up mode (validated to match before submitting) and a show/hide eye-icon toggle on each password field (independent state per field, so showing one doesn't affect the other) via the shared `PasswordVisibilityToggle.jsx` (extracted so `ChangePasswordModal.jsx` below doesn't duplicate it).
- **Change Password** (Settings, its own row right under the title ‚Äî only rendered when `user` is non-null *and* `hasPasswordProvider(user)`, a `SettingsTab.jsx` helper checking `user.identities` for a real `'email'` entry rather than assuming from how the user first signed up, since an account can have both a Google identity and a linked email/password one) ‚Äî `ChangePasswordModal.jsx` is the same modal shell/form style as `AuthModal.jsx` (and reuses its `PasswordVisibilityToggle`), just with only a new-password + confirm-password pair (no email field, no Google/sign-up branches ‚Äî there's already an active session). Calls `AuthContext.jsx`'s `updatePassword(newPassword)`, which is just `supabase.auth.updateUser({ password })` ‚Äî Supabase's own call already requires an active session, so there's no separate "confirm current password" step to build. Shows a success message in place of the form on success rather than auto-closing, mirroring `AuthModal`'s `info` message pattern (e.g. its "check your email" state) for a calm, non-disorienting confirmation.
- **Delete Account** (Settings' Danger Zone, only rendered when `user` is non-null) ‚Äî `SettingsTab.jsx`'s `handleDeleteAccount` asks for a strong confirmation (same severity as Factory Reset's, but explicit that this destroys the Supabase account itself, not just local data), then calls `deleteAccount()` and hard-reloads on success (landing back in guest mode, same as every other Danger Zone action). `deleteAccount()` calls a `delete_user()` Postgres function (`supabase/schema.sql`) via `supabase.rpc(...)` rather than `supabase.auth.admin.deleteUser()` ‚Äî the admin API requires the service-role key, which must never reach the browser. `delete_user()` is `SECURITY DEFINER` (so it can reach `auth.users`) but only ever deletes the *calling* user's own row (`auth.uid()`), and every table's `user_id` FK is `on delete cascade`, so this one call removes the account and all of its Supabase data in one step. **This SQL function must be pasted into the Supabase SQL Editor before the feature works** ‚Äî same manual-apply pattern as the rest of `schema.sql`; there is no migration tooling in this repo that applies it automatically.
- **`ProfileMenu.jsx`** (replaces the earlier `AccountStatus.jsx`) ‚Äî rendered in `App.jsx`'s header (top-right, next to the date/time and the Settings gear icon), visible on every tab. Guest: a plain "Sign in" button (opens `AuthModal`). Signed in: a circular avatar button (the email's first letter) that opens a small dropdown ‚Äî click-outside/Escape-to-close, the same from-scratch pattern as `Select.jsx`. "Sign out" (same `window.confirm` gate as before) is the only entry today; `menuItems` is a plain array specifically so a future entry (e.g. "Account") is just one more array item, not a restructure. Renders nothing while `loading`, same reasoning as the old `AccountStatus.jsx`.

### Cloud sync (Supabase storage provider + migration)

Signing in swaps `storage.js`'s persistence layer from localStorage to Supabase, transparently to every hook ‚Äî no hook file changes. `storage.js` holds `activeProvider` (mutable, defaults to `localStorageProvider`); both it and the new `src/lib/remoteProvider.js` implement the same `get(collection, fallback)` / `set(collection, value)` / `remove(collection)` shape keyed by the existing localStorage key strings (e.g. `'pomodoro_inventory'`), so every `loadX()`/`saveX()` in `storage.js` is unchanged.

- **`remoteProvider.js`** keeps an in-memory `cache` object, always read synchronously by `get()` ‚Äî this is what lets hooks' `useState(() => loadX())` / `useEffect(() => saveX(items), [items])` patterns keep working even though the actual persistence is an async Supabase call. `set()` updates the cache synchronously and fires the upsert in the background (never awaited by the caller). Array collections (`pomodoro_inventory`, `pomodoro_today_tasks`, `pomodoro_activity_log`, `pomodoro_ticks`, `pomodoro_timetable`, `pomodoro_categories`, `pomodoro_void_log`) map to one Supabase table each; `pomodoro_timer_state`/`pomodoro_settings` are one-row-per-user singletons (`onConflict: 'user_id'`). Since `saveX(items)` always passes the *full* authoritative array, `set()` diffs the new array's ids against a `knownIds` cache to compute which ids were removed and issues an explicit `.delete().in('id', ...)` after the upsert ‚Äî an upsert alone never deletes rows.
- **`src/lib/keyCase.js`** ‚Äî converts camelCase JS keys to snake_case Postgres columns and back, with explicit overrides for `diffI`/`diffII` (the generic algorithm can't round-trip these: `diffII` ‚Üí `diff_i_i` generically, but the schema column is `diff_ii`).
- **One-time migration on sign-in** (`storage.js`'s `signInToRemote(userId)`, called by `App.jsx`): snapshots every current localStorage collection, then `remoteProvider.initializeRemoteData()` fetches each Supabase table and merges local into remote with `mergeCollectionById` ‚Äî the same merge function the JSON-import feature already uses (newer `updatedAt` wins; ties/no-timestamp favor the remote/existing side). Singletons use a simpler rule: an existing remote row always wins outright (never clobbered by a fresh guest session's defaults); a local value is only pushed up if no remote row exists yet. `storage.js` only clears localStorage (`clearLocalGuestData()`) *after* `initializeRemoteData` returns successfully ‚Äî a network/RLS error leaves localStorage completely untouched and surfaces an error banner instead, so a failed sync can never lose data.
- **`App.jsx`** is the auth-transition gate: a `dataMode` state machine (`'guest' | 'syncing' | 'remote'`). Guest mode has zero added latency ‚Äî the render tree mounts immediately, before `useAuth`'s initial session check even resolves. Only an actual sign-in transition (detected via a `prevUserIdRef` comparison, so effect re-fires from reference churn don't trigger duplicate work) shows a brief `<SyncingScreen/>` while `signInToRemote()` runs, then remounts the render tree via a `key` prop change so every hook's `useState` initializer re-runs against the now-warm cache. A dismissible banner shows the "synced to your account ‚úì" notice once, and a `syncError` banner on failure; both are reset on every new sign-in/sign-out transition so a stale notice can't reappear on a later session.
  - **`appKey` is its own state, set only from inside the auth effect ‚Äî never derived from `user`/`dataMode` at render time.** An earlier version computed it inline as `dataMode === 'remote' && user ? user.id : 'guest'`, which caused a real bug: signing out flips `user` to `null` (from `useAuth`, a separate context) *before* this component's own effect has run, so that inline expression evaluated to `'guest'` ‚Äî and therefore remounted `AppInner` ‚Äî on the very render where `user` went null, which is strictly *before* `signOutFromRemote()` (only called inside the effect, which fires after render/commit) had switched `activeProvider` back to `localStorageProvider` or cleared `remoteProvider`'s cache. The freshly mounted `AppInner`'s hooks (`useInventory` etc.) read storage during that same premature remount, so the "guest" screen kept showing the just-signed-out account's Supabase data. The fix: `setAppKey('guest')`/`setAppKey(userId)` are called explicitly, only *after* `signOutFromRemote()`/a successful `signInToRemote()` have already run synchronously earlier in the same effect ‚Äî so the remount (and the fresh hooks' storage reads) can never race ahead of the provider switch. Regression-tested in `src/App.test.jsx` (mocks `useAuth` and `remoteProvider` only, so it needs no live Supabase project; confirmed to fail against the old inline-derived `appKey` and pass against the fix).
- **Known limitation, not fully resolved**: `remoteProvider.js`'s singleton fetch has one retry after a 300ms delay (`fetchSingletonTableWithRetry`), defending against a hypothetical transient "no row found" result right after sign-in (e.g. two concurrent sign-ins to the same account). This was added while chasing a since-understood-to-be-unrelated bug (a stale notice-banner state, now fixed) and its necessity against a real race was never conclusively proven ‚Äî it's cheap insurance, not a confirmed fix for a confirmed bug.

### Gotchas

- `vite-plugin-pwa` is wired into `vite.config.js` (`generateSW` strategy, `registerType: 'autoUpdate'`) ‚Äî production builds emit a manifest and service worker, making the app installable and offline-capable. It only activates on `npm run build` + `npm run preview` (or a real deploy); `npm run dev` does not register a service worker unless `devOptions.enabled` is set.
- In-app comments and some identifiers are in Turkish; code style otherwise follows standard React/JS conventions.
- Superseded by the i18n system below: user-facing app text is no longer English-only ‚Äî it's translated per the active language (English/Turkish). What still holds is the *reason* behind the old rule: no language should ever leak into the wrong mode (an English string showing up under Turkish, or vice versa) ‚Äî every user-facing string must go through `t()`/the translation dictionaries, never be hardcoded in a component. In-app code comments stay Turkish/English-mixed as before, unaffected by this.
- Deployed to GitHub Pages via `.github/workflows/deploy.yml` on every push to `main`. The workflow runs `npm run lint` and `npm test` before `npm run build`, so a failing test or lint error blocks the deploy. `vite.config.js`'s `base` is only rewritten to `/pomodoro-app/` when the `GITHUB_PAGES` env var is set (the workflow sets it before `npm run build`), so local `dev`/`build`/`preview` stay unprefixed. GitHub Pages must have its Source set to "GitHub Actions" once in the repo settings (Settings ‚Üí Pages) for the workflow's deploy step to succeed. The build step also forwards `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` from repository secrets ‚Äî Vite only bakes `VITE_`-prefixed env vars present in the process environment at build time into `import.meta.env`, regardless of whether they came from a `.env` file (local) or CI's `env:` block (deploy); without these, the deployed build falls back to `supabaseClient.js`'s `null`-client guest-only path (see Authentication above).

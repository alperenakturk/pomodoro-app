# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm run dev       # start dev server (Vite HMR)
npm run build     # production build
npm run preview   # preview production build locally
npm run lint      # run oxlint (whole project)
npx oxlint src/hooks/usePomodoro.js   # lint a single file
npm test          # run the Vitest suite once
npm run test:watch # run Vitest in watch mode
```

Tests run under Vitest with jsdom (config lives in the `test` block of `vite.config.js`). Hook tests use `@testing-library/react`'s `renderHook` and fake timers to drive the countdown without waiting on real time ŌĆö see `src/hooks/usePomodoro.test.js` for the pattern. Browser-only side effects (audio, notifications) live in `src/lib/alert.js` and are mocked with `vi.mock` in tests, since jsdom doesn't implement `AudioContext`/`Notification`. Pure calculation helpers used by a component (e.g. `countAvailablePomodoros` in `src/lib/pomodoroMath.js`, `isCurrentBlock` in `src/lib/timetable.js`) live in their own `src/lib/*.js` file rather than being exported from the component ŌĆö co-exporting a plain function from a component file breaks Fast Refresh (oxlint's `only-export-components` warns on this).

## Architecture

This is a Pomodoro timer app implementing the Pomodoro Technique (25-min work sessions, short/long breaks, interruption tracking). Stack: React 19, Vite 8, Tailwind CSS v4, oxlint.

### Data flow

All persistent state lives in `localStorage`. `src/lib/storage.js` is the single point of access ŌĆö it exposes typed load/save helpers for its storage keys:

| Key | Purpose |
|-----|---------|
| `pomodoro_inventory` | Activity Inventory ŌĆö the task backlog |
| `pomodoro_today_tasks` | Tasks selected for today with realized/interruption counts |
| `pomodoro_activity_log` | Completed-task records (estimate vs. actual) |
| `pomodoro_ticks` | Lightweight per-pomodoro/interruption events for reports |
| `pomodoro_settings` | User preferences: `cycleLength`, `shortBreakMinutes`, `longBreakMinutes`, `theme`, `chimeStyle` |
| `pomodoro_categories` | User-defined categories (name + color) ŌĆö see Categories below |
| `pomodoro_void_log` | Voided-Pomodoro journal entries (reason, elapsed time) ŌĆö see Void log below |

`patchSettings(partial)` merges into existing settings and must be used instead of `saveSettings` whenever a feature only owns one key of the settings object ŌĆö otherwise it clobbers the others. `exportAllData()` bundles every key for the JSON backup download in `RecordsLog`.

`storage.js` also owns a `pomodoro-data-changed` custom window event. Any write that should notify passive listeners (adding a tick, completing a task) dispatches this event. Components that read from storage but don't receive props ŌĆö `RecordsLog` and `Reports` ŌĆö subscribe with `subscribeToChanges()` to re-read when data changes.

**Backend-readiness prep:** every `loadJSON`/`saveJSON` call goes through an internal `localStorageProvider` object (`get`/`set`/`remove`, keyed by collection) instead of touching `localStorage` directly ŌĆö swapping in a remote backend later means replacing that one object, not every call site. Every normalized record also carries `userId` (always `'local'` for now), `createdAt`, and `updatedAt` ŌĆö old records without them just get `null` timestamps, same `?? null` pattern the rest of normalization already uses. `storage.js`'s own `addActivityRecord`/`addTick`/`addVoidLogEntry` stamp `createdAt`/`updatedAt` at creation (`stampCreated`); `updateActivityRecord` bumps `updatedAt` (`stampUpdated`). Inventory/Today-tasks/Timetable/Categories have no per-record add/update functions in `storage.js` ŌĆö hooks `saveX` the whole array instead ŌĆö so those only gain real timestamps once such granular functions exist; this is schema prep, not a working sync layer (still no async I/O, realtime channel, or auth).

### Hook layer

Five hooks own all mutable state:

- **`useInventory`** ŌĆö CRUD for the Inventory list; auto-saves to localStorage on every change. Also owns `combineItems` (Rule 5: merge 2+ small tasks into one, summing estimates).
- **`useTodayTasks`** ŌĆö manages today's task list: realized pomodoro count, internal/external interruption counters, active task selection, re-estimates (`reestimateTask`, up to `reestimate1`/`reestimate2`). Calling `finishTask` writes a record (with `diff`/`diffI`/`diffII`) to the activity log and marks the task done.
- **`usePomodoro`** ŌĆö pure timer state machine (work ŌåÆ shortBreak/longBreak cycle). After every `cycleLength` completed pomodoros (user-configurable, default 4) it automatically switches to a long break. It writes a tick on each completed work session and interruption. Work/short-break/long-break lengths are all user-configurable (`workMinutes`/`shortBreakMinutes`/`longBreakMinutes`, defaults 25/5/15) via `setWorkMinutes`/`setShortBreakMinutes`/`setLongBreakMinutes`, matching the same `patchSettings`-on-change pattern as `setCycleLength`. Short/long break clamp to hard bounds (3ŌĆō10 and 15ŌĆō60, exported as `SHORT_BREAK_MIN/MAX`/`LONG_BREAK_MIN/MAX`) with a non-blocking `SettingsTab.jsx` hint when the value exceeds methodology.md's recommended max (5 / 30) ŌĆö still fully accepted past that point. Work duration has no recommended-range concept at all (per methodology, 25 minutes just *is* the technique) ŌĆö only the sane absolute bounds `WORK_MIN`/`WORK_MAX` (1ŌĆō180) apply, and `SettingsTab.jsx` instead shows an informational note whenever it differs from `DEFAULT_WORK_MINUTES` (25), framed as a deliberate deviation rather than a recommendation to revert. An idle (not-running) session's displayed countdown stays in sync with its type's current setting via a dedicated effect ŌĆö so changing a duration takes effect the next time Start is pressed, not just after one extra full cycle ŌĆö but a session that's actually running is never resized out from under the user (the effect no-ops while `isRunning`), and so is a *paused* session: a `pausedRef` (plain `useRef`, not state ŌĆö it doesn't need to trigger a render) is set `true` only by `pause()` and cleared by every fresh-session transition (`start`/`completeWork`/`completeBreak`/`voidPomodoro`/`skipBreak`/`switchSession`), and the resync effect no-ops while it's set, so a duration change made while paused can't silently overwrite the paused elapsed time back to a fresh full duration. `Timer.jsx` mirrors these three values itself (via the same `pomodoro` prop spread) to compute the ring's progress arc, since it has no other way to know each session type's full duration.
- **Drift-resistant countdown** ŌĆö the countdown is **not** a per-tick `secondsLeft - 1` decrement; it's driven by an absolute end timestamp. `endAtRef` (a `useRef`, not state) holds `Date.now() + secondsLeft * 1000` the moment a session starts or resumes (set in `start()`); the once-a-second `setInterval` effect and the on-mount restore-from-`localStorage` path both *recompute* `secondsLeft` as `Math.round((endAtRef.current - Date.now()) / 1000)` rather than trusting the previous value. This means a throttled/backgrounded tab, or the page being closed and reopened entirely (restored from `pomodoro_timer_state`'s persisted `endAt` field), always lands on the true remaining time instead of compounding missed ticks into drift. See `src/hooks/usePomodoro.test.js` for the remount-across-a-large-time-gap regression test.
- **Auto-start** ŌĆö two independent, default-off toggles: `autoStartBreaks` (a completed Pomodoro's break begins immediately instead of waiting for Start) and `autoStartPomodoros` (a finished break's next work session begins immediately). Both are consulted at the end of `completeWork`/`completeBreak` respectively; `skipBreak` also honors `autoStartPomodoros` (skipping is still "a break ending," just early) ŌĆö manual `switchSession` (the session-switch pills) deliberately does not auto-start either way, since that's an explicit user choice of where to jump, not a natural transition.
- **Sound volume** ŌĆö `soundVolume` (0ŌĆō100, default 100) is applied globally via `alert.js`'s own module-level `setVolume(percent)`/`volume` (0ŌĆō1), which every `playTone` call scales its peak gain by ŌĆö so `usePomodoro` only needs a `useEffect` calling `setAlertVolume(soundVolume)` on change, rather than threading a volume argument through every `playChime`/`playPing`/`playTaskCompleteChime` call site. `alert.js` floors the scaled peak at `0.0001` rather than letting it reach exactly `0`, since Web Audio's `exponentialRampToValueAtTime` throws on a zero target ŌĆö reachable for real at the volume slider's minimum.
- **Ambient background sound** ŌĆö `ambientSound` (`alert.js`'s `AMBIENT_SOUNDS`: `'none'` default, `'ticking'`, `'rain'`, `'cafe'`, `'whiteNoise'`) plays for as long as `ambientSound !== 'none' && isRunning && sessionType === 'work'` all hold, via one `useEffect` keyed on exactly those three values ŌĆö pause/void/completion/break all naturally stop it since each flips at least one of them. `alert.js`'s `startAmbientSound(kind)`/`stopAmbientSound()` route to one of two implementations: `'ticking'` keeps the original discrete-click approach (a `setInterval` re-triggering a short `playTone` once a second ŌĆö its volume is read fresh every tick, no live-update needed); `'rain'`/`'cafe'`/`'whiteNoise'` share a single noise-buffer audio graph (a looping `AudioBufferSourceNode` filled with random samples ŌåÆ an optional `BiquadFilterNode` ŌåÆ a `GainNode` ŌåÆ destination), differing only in which filter (if any) shapes the same raw noise ŌĆö unfiltered for white noise, band-passed for rain, low-passed (plus occasional random high-pitched "clink" blips) for cafe. All are simple Web Audio approximations, not recordings, consistent with this file's no-external-assets approach. Since these beds are *continuous* (unlike the one-shot tones), `setVolume()` also live-updates the currently-playing bed's gain node, not just the module-level `volume` multiplier the one-shot sounds read fresh each time. A legacy boolean `tickingSoundEnabled` setting migrates automatically (`storage.js`'s `normalizeAmbientSound`): `true` ŌåÆ `'ticking'`, `false` ŌåÆ `'none'`, only when no `ambientSound` value has been saved yet.
- **`useTimetable`** ŌĆö today's planned time blocks (`Timetable.jsx`); blocks are stamped with the date they were created and pruned on load if they're from a previous day.
- **`useCategories`** ŌĆö CRUD for user-defined categories (name + color, from `CATEGORY_COLORS` in `constants.js`). No cascade delete: `removeCategory` just drops the category, and anything referencing its id afterward fails to resolve it and falls back to "no category"/"Uncategorized" ŌĆö the same fallback legacy/free-text `type` data already needed, so it's free.

### Component wiring

`App.jsx` instantiates `useInventory`, `useTodayTasks`, `usePomodoro`, and `useCategories`, then passes their data down as props ŌĆö `Timer` and `SettingsTab` are both purely presentational and share the one `usePomodoro` instance (`SettingsTab` needs `cycleLength`/`chimeStyle` setters that live in the same hook Timer renders from, so it can't be timer-local anymore). `useTimetable` is instantiated inside `TodoToday` (not `App`) so its block totals can feed `AvailablePomodoros`' suggested-hours button. The hooks never talk to each other directly ŌĆö `App`/`TodoToday` bridge them: when a task from Inventory is sent to Today it stores the source `inventoryId`, `categoryIds`, and `notes` on the today-task, and when that task is finished `App` also removes it from Inventory. Marking a task done is deliberately decoupled from the timer ŌĆö finishing a task early doesn't stop a running Pomodoro (the remaining time is still there for overlearning). A "Check to bottom" setting (`checkToBottom`, default off, managed in `App.jsx` alongside `theme`/`onboardingDismissed` since it's not owned by any one hook) additionally calls `useTodayTasks`' `moveTaskToEnd(id)` right after `finishTask` when enabled ŌĆö `useTodayTasks` itself stays unaware of Settings, same separation as everywhere else in this bridging layer. Moving to the end of the *array* lands a task at the end of whichever of planned/Unplanned-&-Urgent `TodoToday.jsx` filters it into (filtering preserves relative order), which reads as "bottom of its own section," not "bottom of everything."

`RecordsLog` reads storage directly on mount and on `pomodoro-data-changed` events, but takes `categories` as a prop (from `categoriesApi.categories`) rather than loading it itself ŌĆö it's read-only reference data needed for display/filtering, same reasoning as `Reports`' `todayTasks` prop below. `Reports` does the same for `ticks`/`activityLog`, but also takes two props ŌĆö `todayTasks={todayApi.tasks}` (for its "active task" count: `todayTasks` isn't date-stamped and `saveTodayTasks` doesn't dispatch `pomodoro-data-changed`, so reading it from storage on that event would show stale data) and `categories` (for the category-breakdown chart). `Reports` also renders `DayReview` (an end-of-day summary modal, unchanged by the metrics redesign below).

### Inventory row controls

`Inventory.jsx`'s `InventoryRow` has three independent, easy-to-conflate controls: the checkbox is Rule 5 combine-selection (`selected`, local `Set` state in `Inventory`, not persisted ŌĆö purely a UI selection, unrelated to `done`), the circular button is the actual `done` toggle (`toggleDone`, persisted, and the only thing the strikethrough style is bound to), and the "U" badge is `item.unplanned`. A selected row gets a `bg-tomato/10`/`border-tomato/30` tint (same token pairing as the active-row highlight in `TodoToday.jsx`/`Timetable.jsx`) so combine-selection has visible feedback beyond the "Combine" banner appearing. `unplanned` is set once at add-time but is also editable in edit mode (a "U" toggle button identical to the add-form's, persisted via `updateItem`) ŌĆö unlike categories/deadline/notes, it previously had no way to be changed after creation.

### Categories

Replaces what used to be a free-text `type` field on Inventory/Today-task/Records with a `categoryIds` array (multi-select tags, not a single pick) into `useCategories`' list. Assignment uses `CategoryTagPicker.jsx` (checklist-style multi-select dropdown); `CategorySelect.jsx` is a separate, single-pick dropdown kept only for Records Log's category *filter* (filtering by one category at a time is still the simpler, appropriate UX there ŌĆö it was never converted to multi-select). Both are from-scratch dropdowns like `Select.jsx` but with a leading color-dot per option, built as their own components so the existing sound-style `Select.jsx` isn't put at risk. `CategoryManager.jsx` (rendered in the Settings tab) is the create/edit/delete UI, using a curated 8-color palette (`CATEGORY_COLORS` in `constants.js`) rather than reusing `tomato`/`amber`, which already carry other meanings elsewhere (work/danger, break/"took less") that a category swatch in the same color would collide with. A task's pomodoros count fully toward EACH of its assigned categories in the Reports breakdown ŌĆö not split ŌĆö see `pomodorosByCategory()` in `reportsMath.js`.

Migration: `storage.js`'s `normalizeCategoryIds()` wraps a legacy single `categoryId` (string or `null`) into a one-element array (or `[]`), so old single-select data loads without any explicit migration step ŌĆö normalize functions rebuild an explicit object rather than spreading, so the old key is simply never read again.

Every task/record also gained a `notes` field (long-text description) ŌĆö Inventory already had one pre-existing (upgraded from a single-line input to a `<textarea>`); Today's Tasks and Activity Log records are new. Displayed collapsed behind a ­¤ōØ toggle in list views (`Inventory.jsx`, `TodoToday.jsx`'s `TaskRow`, `RecordsLog.jsx`'s `RecordRow`) so it doesn't clutter the row ŌĆö expands on click.

The `pairWith` field (a lightweight "pairing with" note, no real-time sync) has been removed entirely ŌĆö categories now cover that use case. Old records/tasks with a legacy `type` or `pairWith` value load fine; `storage.js`'s normalize functions simply don't read those keys anymore, so they're silently dropped rather than migrated.

### Fullscreen Focus Mode

`Timer.jsx` renders one shared JSX tree for both the normal card view and the fullscreen minimal view, toggling extra chrome on/off with `{!isFullscreen && (...)}` guards (session-switch pills, the "?" keyboard-shortcuts button, `UnplannedCapture`) rather than maintaining two separate component trees. `isFullscreen` is driven only by the native `fullscreenchange` event (not set optimistically on click) ŌĆö `requestFullscreen()` requires a user gesture and can still be rejected in some contexts, so relying on the event keeps the UI honest about actual fullscreen state regardless of how it was entered/exited (button, `F`, or the browser's own Escape-to-exit). Pause (like Void/Skip) is deliberately **not** gated behind `!isFullscreen` ŌĆö it's a core timer control, not extra chrome, so it stays available even in Fullscreen Focus Mode.

Keyboard shortcuts: `F` toggles fullscreen; `Space` starts the timer when idle and **toggles Pause/Resume when running** (previously Space only ever started); `Escape` closes the Keyboard Shortcuts modal if it's open, else opens the Void-reason prompt (see below) when a Pomodoro is running, cancels that prompt if it's already open, and does nothing at all while fullscreen (the browser's own Escape-exits-fullscreen handling takes over ŌĆö the handler explicitly no-ops there so it doesn't *also* open the void prompt underneath). `?` opens the Keyboard Shortcuts modal (see below). There is deliberately no keyboard shortcut for manually switching Focus/Short Break/Long Break (removed `1`/`2`/`3`) ŌĆö the session-switch pills are still click-only. There is also no shortcut for finishing a Pomodoro early ŌĆö that control no longer exists at all (see Pause / Resume below).

### Void log

Voiding a Pomodoro (Rule 1) opens an inline panel (`Timer.jsx`, replacing the old bare `window.confirm`) with an optional free-text reason, combining confirmation and reason-capture into one step. `usePomodoro.js`'s `voidPomodoro(reason)` computes elapsed time itself (`DURATIONS.work - secondsLeft`) and reports `{ reason, elapsedSeconds }` via a new `onVoid` callback ŌĆö mirroring the existing `onWorkComplete`/`onInterruption` bridge pattern ŌĆö since the hook has no knowledge of the active task; `App.jsx`'s `onVoid` implementation looks up the active task/its categories and writes the entry via `addVoidLogEntry()`.

### Pause / Resume

Replaces the old "Finish Pomodoro" (finish-early) button entirely ŌĆö that control, and all of its logic (`finishEarly`, its confirm dialog, its `E` shortcut), has been removed. Per `docs/methodology.md`'s Rule 2, a Pomodoro is meant to always ring; this is a distinct, more transparent deviation from that rule (documented in both `usePomodoro.js`'s `pause()` comment and methodology.md itself): users sometimes need to genuinely step away mid-Pomodoro without it rising to Rule 1's "void it" threshold, so Pause holds the session in place ŌĆö no X recorded, no time lost ŌĆö rather than completing it early (the old behavior) or discarding it.

- **`usePomodoro.js`'s `pause()`** stops the countdown (freezing `secondsLeft` at its current value via the same `endAtRef`-vs-`Date.now()` computation the interval tick uses), clears `isRunning`, increments a live `pauseCount` state (reset to 0 on every fresh-session transition ŌĆö same set of functions listed under "Drift-resistant countdown" above), and writes a `'pause'`-typed tick via the existing `addTick()` ŌĆö no new storage collection needed, since `pomodoro_ticks`' `type` field was already an unrestricted string. **Resuming reuses `start()` directly** (no separate `resume()`) ŌĆö `start()` always computes `endAtRef` from the *current* `secondsLeft`, which `pause()` deliberately leaves untouched, so it's correct for both a fresh start and a resume.
- **`Timer.jsx`** shows a "Resume" label instead of "Start" whenever the session is idle with a partial `secondsLeft` (`0 < secondsLeft < sessionDuration`, i.e. can only be reached via `pause()`). The Pause button itself renders whenever `isRunning`, for both work and break sessions (unlike Void/Skip, which are session-type-specific). The live `pauseCount` renders as a small, subtle line under the controls, the same visual weight as the interruption counters, once it's > 0.
- **Start is disabled** when the upcoming session is work and there's no active task (`isWork && !activeTask`), with a `title` explaining why ŌĆö breaks remain always startable. This closes the previous gap where a work Pomodoro could run fully untracked (no task to attribute realized pomodoros/interruptions to). The `Space` shortcut respects the same guard.
- **Reports** surfaces pause history via `reportsMath.js`'s `pausesByDate(ticks, dates)` (per-day counts, oldest-first) and `avgPausesPerDay(ticks, dates)` ŌĆö a new "Pause Trends" section (`PauseTrendsSection`, slotted into `SECTIONS` right after Interruption Trends) shows an avg-per-day stat, a this-week-vs-last-week trend arrow (same `trendDirection`/`goodDirection="down"` pattern as Interruption Trends), and a per-day bar list capped at the last 14 days in the selected period. `TodaySection` also gained a 5th "Pauses today" `Stat` box (today-vs-yesterday trend), accepting the resulting `grid-cols-2` layout asymmetry (5 boxes, last row has one) as a minor cosmetic tradeoff.

### Picture-in-Picture mini timer

`Timer.jsx` uses the Document Picture-in-Picture API (`window.documentPictureInPicture`, feature-detected via `isPipSupported()` in `src/lib/pip.js` ŌĆö the toggle button simply doesn't render when unsupported, e.g. Safari; `document.title`'s existing countdown remains the universal fallback). The PiP window's content is a `ReactDOM.createPortal` into `pipWindow.document.body`, so it re-renders in lockstep with the main countdown automatically ŌĆö no second timer, nothing to keep in sync manually. `copyStylesToWindow()` (also in `pip.js`) copies every stylesheet from the main document into the new (otherwise blank) PiP document, the pattern the API's docs recommend, so the portal content picks up the app's existing Tailwind classes/color tokens; the portal's own root div also re-applies the `.light` theme class (Timer now receives `theme` as a prop from `App.jsx` for this, since it has no other reason to know about theme). `pipWindow` state (open Window object, or `null`) is driven by the PiP window's own `pagehide` event rather than set optimistically, so it can't drift out of sync whether the user closes it from the mini window, the toggle button, or by switching away ŌĆö same reasoning as `isFullscreen` above. Deliberately read-only content (no interruption buttons, no controls) ŌĆö it exists purely so the countdown stays visible when the user is in another tab/app, not as a second control surface.

This is deliberately a simple daily journal, not a metric: `Reports.jsx`/`reportsMath.js` never read `pomodoro_void_log` at all. `RecordsLog.jsx` shows entries in an unobtrusive "Voided Pomodoros" section (own `VoidLogRow`, capped at 5 most recent) that is **not** affected by the Records Log date/category filters ŌĆö those filters exist for the Activity Log's aggregation-adjacent use, not this.

### Reports metrics

`Reports.jsx` is organized around `src/lib/reportsMath.js` (pure, unit-tested date/aggregation helpers ŌĆö `effectiveDiff`, `datesForPeriod`/`datesForThisWeek`/`datesForLastWeek`/etc., `estimationBreakdown`, `avgInterruptionsPerTask`, `trendDirection`). Per methodology, metrics favor trends and per-task ratios over cumulative totals:

- **Today** ŌĆö fixed today-vs-yesterday comparison (pomodoro count, task count, interruptions, pauses), independent of the time filter below ŌĆö except for its "Total focus time" card, which is the one deliberate exception: `reportsMath.js`'s `totalFocusMinutes(ticks, dates, workMinutes)` multiplies the *selected period's* completed-Pomodoro tick count by the current `workMinutes` setting (passed down from `usePomodoro` via `App.jsx` as a prop), formatted H:MM by `formatFocusDuration()`. Ticks record only that a Pomodoro completed, not how long it actually ran, so this is an approximation ŌĆö "N pomodoros at today's current work length" ŌĆö the same assumption `AvailablePomodoros`' capacity estimate already makes elsewhere, not a literal historical total that accounts for `workMinutes` having changed over time. This card carries a small "Ōōś" next to its label (native `title` tooltip, `reports.totalFocusTimeTooltip`) spelling out that approximation explicitly, since changing the work-duration setting silently shifts what this number means for past periods too.
- **Estimation Accuracy** ŌĆö overestimated/underestimated counts and a per-task diff chart, both scoped to the top time filter (Today/Week/Month/Year, default Week); plus a fixed this-week-vs-last-week avg-error trend.
- **Interruption Trends** ŌĆö average interruptions *per finished task* (not a raw total ŌĆö the deliberate correction over the old design), scoped to the time filter, plus a fixed week-over-week trend and a per-task breakdown list.
- **Pause Trends** ŌĆö see Pause / Resume above; sits right after Interruption Trends in the `SECTIONS` stepper, since both are "what happened during work" self-observation metrics.
- **Pomodoros by Category** ŌĆö `pomodorosByCategory()` sums each finished record's `real` grouped by `categoryId`, scoped to the time filter, rendered as a horizontal bar per category colored with that category's own color (an "Uncategorized" bucket, sage-colored, catches null/deleted-category records). This only counts pomodoros from *finished* tasks ŌĆö ticks (the raw pomodoro events) don't carry a category, so there's no way to attribute an in-progress task's pomodoros to a category without adding new tracking.
- **Long-term** ŌĆö collapsed by default; the 13-week heatmap plus a month/quarter summary.

`effectiveDiff` also lives here now (moved out of duplicate copies previously in `Reports.jsx` and `DayReview.jsx`); `DayReview.jsx` imports it from `reportsMath.js`.

**Empty states** ŌĆö three distinct "nothing to show" messages, not to be confused with each other:
- **No data anywhere** (`ticks.length === 0 && activityLog.length === 0`) ŌĆö replaces the entire period-filter row and all sections with a single calm message ("Complete a few Pomodoros to see your trends here."); there's nothing meaningful to filter yet.
- **`hasNoHistoryYet()`** (existing, unchanged) ŌĆö there *is* data, but every tick/record is dated today, so Today/Week/Month/Year would all show the identical result. An italic hint above the sections, not a replacement.
- **Per-period empty** (new) ŌĆö the selected period specifically has zero matching records (most commonly: "Today" selected on a day nothing's finished yet), even though other periods have data. `EstimationAccuracySection`/`InterruptionTrendsSection`/`CategoryBreakdownSection` each independently check their own `recordsInDates(activityLog, datesForPeriod(period))` and substitute a shared `t('reports.noDataForPeriod')` message for their entire content when empty, rather than rendering stat boxes full of zeros next to an empty chart (which is what looked broken before this fix). `EstimationAccuracySection` didn't have any such check previously ŌĆö `InterruptionTrendsSection` and `CategoryBreakdownSection` already had the equivalent logic under differently-worded keys, now unified onto the shared key.

Opened via a small "?" icon button next to Fullscreen/PiP (hidden in fullscreen, same gate as PiP) or by pressing `?` itself ŌĆö both handled in `Timer.jsx`'s existing keydown listener, which also closes it on `Escape` (checked before the void-prompt's own Escape handling, so pressing Escape with the modal open never also opens the void prompt underneath). ### Tab layout

`App.jsx` renders three tabs via `TabNav` (`activeTab` state: `timer` | `planning` | `reports` | `settings` ŌĆö `settings` is a valid value but reachable only via the header's gear icon, not through `TabNav` itself) to keep the Timer screen distraction-free per methodology ŌĆö no Inventory/Records/Reports/Settings content should be visible while a Pomodoro is running. **All four tab panels stay mounted; only the active one is toggled visible with Tailwind's `hidden` class** (not conditionally rendered). - **Timer tab** ŌĆö `Timer.jsx`: ring + state label, current active task, Start (disabled for a work session with no active task) / Pause┬ĘResume / Void / Skip-break, interruption buttons (internal/external, gated on `isRunning`), and `UnplannedCapture` (add-only input, no list ŌĆö jot it down and keep working).
- **Planning tab** ŌĆö `Inventory` + `TodoToday` (which itself includes `AvailablePomodoros`, `Timetable`, the today-task list, and the full Unplanned & Urgent box with its list). `TodoToday.jsx`'s header row also has a "..." `TaskListMenu` (same click-outside dropdown pattern as `Select.jsx`/`ProfileMenu.jsx`) with two bulk actions ŌĆö "Clear finished tasks" (`useTodayTasks`' `clearFinishedTasks`, disabled when nothing's done) and "Clear all tasks" (`clearAllTasks`, disabled when the list is empty) ŌĆö each still gated behind its own `window.confirm`. Both only ever touch the array in memory/storage; neither is timer-aware.
- **Reports tab** ŌĆö `Reports` (which renders `DayReview` internally) + `RecordsLog` (which also has its own date-picker + category filter, combined with AND logic; unfiltered view caps at the 8 most recent records, filtered views show every match). No longer has export buttons ŌĆö see Data import & export below.
- **Settings tab** ŌĆö `SettingsTab.jsx`: cycle length, work/short/long break duration, auto-start toggles, chime/sound + volume + ambient-sound picker, theme toggle, "Check to bottom", and language (see Internationalization below). ### Data import & export

`DataTransfer.jsx` (Settings tab, titled "Import & Export Data") holds all four data-transfer options in one place ŌĆö JSON Export, CSV Export, JSON Import, CSV Import, each with a one-line description of what it does/doesn't bring back. Export used to live as CSV/JSON buttons in `RecordsLog.jsx`'s header; moved here (this component was previously `DataImport.jsx`, import-only) so a user managing their data isn't split across two tabs. Export itself is unchanged ŌĆö `exportAllData()`/`activityLogToCSV()` + `downloadFile()`, same as before, just called from here.

Import is backed by pure logic in `src/lib/importData.js` and orchestration in `storage.js`'s `importBackup(data, mode)` / `importActivityLogCSV(rows, categories, mode)`. Both file types share the same flow: pick a file ŌåÆ parse + validate the structure (`validateBackupShape`/`validateActivityCSV` ŌĆö reject with an inline error and touch nothing on a bad file) ŌåÆ an inline "Replace all data" / "Merge with existing" / "Cancel" choice appears ŌåÆ `window.confirm` (same severity/style as Danger Zone) ŌåÆ write ŌåÆ `window.location.reload()` (same reason as the Danger Zone reload above).

- **JSON (full backup)** ŌĆö `validateBackupShape` checks the parsed object's known array fields (`inventory`, `todayTasks`, `activityLog`, `ticks`, `timetable`, `categories`, `voidLog`) are actually arrays, and `settings` (if present) is a plain object. **Replace** writes every included collection (and `settings`) as-is, wholesale. **Merge** resolves each array collection by `id` via `mergeCollectionById` (newer `updatedAt` wins; a record with no `updatedAt` counts as older than one that has a real timestamp; if neither side has one, the existing record wins rather than flipping a coin) ŌĆö and deliberately leaves `settings` untouched even in merge mode, since reconciling data isn't the same thing as silently overwriting the user's current theme/language/cycle-length preferences with whatever the file happened to have at export time.
- **CSV (Records/Activity Log only)** ŌĆö `export.js`'s `CSV_COLUMNS` const is exported so `validateActivityCSV` can check an imported file's header row matches it exactly (same columns, same order) rather than accepting "any CSV." `parseCSV` is a small hand-rolled parser (quoted fields, doubled `""`, embedded newlines in `notes`) mirroring `escapeCSVField`'s output format. CSV rows carry no `id` (see `CSV_COLUMNS`), so merge can't use the JSON path's id+`updatedAt` logic ŌĆö `mergeActivityRecordsByNaturalKey` instead matches on the `(date, time, activity)` triple: a match means "skip, keep the existing record" (no reliable way to tell which is newer without an id or a CSV-side `updatedAt`), a non-match gets appended as a fresh record. `csvRowsToActivityRecords` resolves the CSV's "; "-joined category *names* back to ids via a lookup against the current `categories` list ŌĆö a name that no longer resolves (renamed/deleted/typo) is silently dropped from that record's tags, the same graceful-degradation pattern used everywhere else a category reference fails to resolve.

### Empty states & onboarding

Per-screen empty-state copy (Inventory, Today's Tasks, Records Log ŌĆö see their `emptyState`/`noRecordsEmpty` translation keys) is deliberately calm and action-oriented rather than a blank list or a generic "no data" ŌĆö see Reports metrics above for its three-tier empty-state handling specifically.

**First-launch welcome card** (Timer tab only) ŌĆö shown when `pomodoro_settings.onboardingDismissed` is `false` *and* every collection App.jsx can see is empty (`inventoryApi.items`, `todayApi.tasks`, `categoriesApi.categories`, plus a direct `loadActivityLog()`/`loadTicks()` check for completeness). Timer's "no active task" state also gained a `t('timer.goToPlanningButton')` link (`onGoToPlanning` prop ŌåÆ `setActiveTab('planning')`) ŌĆö both are hidden in Fullscreen Focus Mode, consistent with that mode showing only the methodology-clean core controls.

**Replaying the welcome card** ŌĆö Settings' About category has a "Show welcome message again" button (`onReplayWelcome` prop, wired to `App.jsx`'s `replayWelcome()`). Small, calm animation + sound pairs mark two moments ŌĆö deliberately not gamified (no confetti/scores):

### Internationalization (i18n)

Supports English and Turkish. Deliberately hand-rolled rather than a library (e.g. i18next): two locales, no lazy-loading needs, and no plural rules beyond one spot (the Reports heatmap tooltip, handled with explicit `heatmapTooltipOne`/`heatmapTooltipOther` keys rather than a general plural engine) ŌĆö a library would add a dependency and an abstraction layer for features this app doesn't use.

- **`src/lib/i18n/en.js` / `tr.js`** ŌĆö flat-per-namespace nested dictionaries (`timer.start`, `inventory.deleteConfirm`, etc.), kept in sync by hand since both are small.
- **`src/lib/i18n/index.js`** ŌĆö `detectBrowserLanguage()` (Turkish if `navigator.language` starts with `tr`, else English), `resolveLanguage(settingsLanguage)` (uses the explicit setting if present, otherwise auto-detects ŌĆö so auto-detection keeps re-running every load until the user picks a language), `translate(language, key, vars)` (dot-path lookup + `{{var}}` interpolation, falling back to English then to the raw key if missing), and `formatDateLocalized(isoDate, localeTag)` for locale-aware date display (parses `'YYYY-MM-DD'` manually into `new Date(y, m-1, d)` rather than `new Date(isoString)`, which parses as UTC and can roll back a day in negative-offset timezones).
- **`src/lib/i18n/LanguageContext.jsx` + `src/hooks/useTranslation.js`** ŌĆö a React Context is the one deliberate exception to this app's usual prop-drilling-from-`App.jsx` pattern (see Component wiring above): translation strings are needed by nearly every leaf component, so drilling `t` through every intermediate component would be pure boilerplate, unlike theme/categories which only a handful of top-level components need. `LanguageProvider` wraps `<App />` in `main.jsx`; any component calls `useTranslation()` directly for `{ t, language, setLanguage, localeTag }`. Split into two files (context object / provider component / hook) purely to satisfy oxlint's `only-export-components` ŌĆö same reasoning as `pomodoroMath.js`/`timetable.js` being split out from their components.
- **Settings persistence**: `pomodoro_settings.language` defaults to `null` ("not yet explicitly chosen" ŌĆö `resolveLanguage` auto-detects in this case); picking a language in Settings' now-functional Language row (previously a "Coming soon" placeholder) calls `setLanguage`, which persists via `patchSettings` and immediately re-renders every consumer through the Context.
- **`usePomodoro`** is the one exception that does *not* use the Context ŌĆö as a hook (not a component) instantiated once in `App.jsx`, it takes `t` as an options parameter (same bridge-callback pattern as `onWorkComplete`/`onInterruption`/`onVoid`) for its browser-notification strings (`notify(...)` calls in `completeWork`/`completeBreak`). It defaults to a plain `translate('en', ...)` call when `t` isn't passed, so it stays usable standalone (e.g. in tests that don't wrap it in `LanguageProvider`).
- **`reportsMath.js`'s `pomodorosByCategory`** takes an optional `uncategorizedLabel` parameter (default `'Uncategorized'`, for backward-compatible tests) rather than hardcoding the bucket name, since it's a pure lib function with no access to `t()` ŌĆö `Reports.jsx` passes `t('reports.uncategorized')` explicitly.
- **`CATEGORY_COLORS`** (`constants.js`) gained a `key` field per color (`teal`, `plum`, etc.) ŌĆö `CategoryManager.jsx` looks up `t(\`categoryColors.${c.key}\`)` for the swatch tooltip/aria-label instead of the English `name` field, which remains as a dev-facing fallback label.
- **`common.appTitle`** is the one deliberately *un*-translated string in either dictionary ŌĆö both `en.js` and `tr.js` hold the identical literal `'Pomodoro Technique'`, treated as a fixed brand/technique name (per methodology.md's trademark-avoidance note) rather than UI copy. Don't "fix" the Turkish value to a translated form; the duplication across both files is intentional.

### Authentication (Supabase)

Accounts are optional ŌĆö this is additive on top of the existing localStorage-only guest mode, never a replacement for it. `storage.js` itself is untouched by any of this; auth is session state only so far. `supabase/schema.sql` has the full Postgres schema + RLS policies for the eventual data migration.

- **`src/lib/supabaseClient.js`** ŌĆö reads `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` from env (`.env`, gitignored; `.env.example` is the tracked template). Exports `supabase` as `null` when either var is missing, rather than throwing ŌĆö so a checkout without a configured `.env` still runs in guest mode instead of crashing at import time.
- **`src/lib/auth/AuthContext.jsx` + `src/hooks/useAuth.js`** ŌĆö a second Context, structured identically to `LanguageContext`/`useTranslation` (same context-object/provider/hook file split, same reasoning: auth state is needed from deeply nested places like the header's account menu without prop-drilling). `AuthProvider` wraps `<App />` in `main.jsx`, outside `LanguageProvider` (no dependency either way). Exposes `{ user, loading, signInWithGoogle, signInWithEmail, signUpWithEmail, signOut, deleteAccount }`; `user` is `null` for guests and `loading` is only ever true for the brief window before the initial `getSession()` call resolves. All five auth actions no-op with a `NOT_CONFIGURED_ERROR` when `supabase` is `null`.
- **Session sync**: `supabase.auth.onAuthStateChange` (not just the one-time `getSession()` call) keeps `user` current ŌĆö this is also what picks up the session Supabase's client establishes automatically after an OAuth redirect lands back on the app (`detectSessionInUrl` is on by default), so there's no separate "handle the callback" route/page to build.
- **OAuth redirect URL**: built at call time as `window.location.origin + import.meta.env.BASE_URL` (`AuthContext.jsx`'s `signInWithGoogle`) rather than hardcoded, so it resolves to `http://localhost:5173/` in dev and `https://<user>.github.io/pomodoro-app/` on GitHub Pages automatically (`BASE_URL` reflects `vite.config.js`'s `base`, see Gotchas below). **Both exact URLs must be added to the Supabase project's Auth ŌåÆ URL Configuration (Site URL / Additional Redirect URLs) in the dashboard** ŌĆö that allow-list is project configuration, not something any client-side code can set.
- **`AuthModal.jsx`** ŌĆö modeled on `DayReview.jsx`'s modal shell (fixed backdrop, centered card, focus moved in on open/restored on close) for visual consistency with the app's other modals. Google button + email/password form (toggling between sign-in/sign-up) + a "Continue without an account" button that just calls `onClose` ŌĆö dismissing it is a no-op with respect to app state, since guest mode was already the running mode. The email/password form has a "Confirm password" field in sign-up mode (validated to match before submitting) and a show/hide eye-icon toggle on each password field (independent state per field, so showing one doesn't affect the other) via the shared `PasswordVisibilityToggle.jsx` (extracted so `ChangePasswordModal.jsx` below doesn't duplicate it).
- **Change Password** (Settings, its own row right under the title ŌĆö only rendered when `user` is non-null *and* `hasPasswordProvider(user)`, a `SettingsTab.jsx` helper checking `user.identities` for a real `'email'` entry rather than assuming from how the user first signed up, since an account can have both a Google identity and a linked email/password one) ŌĆö `ChangePasswordModal.jsx` is the same modal shell/form style as `AuthModal.jsx` (and reuses its `PasswordVisibilityToggle`), just with only a new-password + confirm-password pair (no email field, no Google/sign-up branches ŌĆö there's already an active session). Calls `AuthContext.jsx`'s `updatePassword(newPassword)`, which is just `supabase.auth.updateUser({ password })` ŌĆö Supabase's own call already requires an active session, so there's no separate "confirm current password" step to build. Shows a success message in place of the form on success rather than auto-closing, mirroring `AuthModal`'s `info` message pattern (e.g. its "check your email" state) for a calm, non-disorienting confirmation.
- **Delete Account** (Settings' Danger Zone, only rendered when `user` is non-null) ŌĆö `SettingsTab.jsx`'s `handleDeleteAccount` asks for a strong confirmation (same severity as Factory Reset's, but explicit that this destroys the Supabase account itself, not just local data), then calls `deleteAccount()` and hard-reloads on success (landing back in guest mode, same as every other Danger Zone action). `deleteAccount()` calls a `delete_user()` Postgres function (`supabase/schema.sql`) via `supabase.rpc(...)` rather than `supabase.auth.admin.deleteUser()` ŌĆö the admin API requires the service-role key, which must never reach the browser. `delete_user()` is `SECURITY DEFINER` (so it can reach `auth.users`) but only ever deletes the *calling* user's own row (`auth.uid()`), and every table's `user_id` FK is `on delete cascade`, so this one call removes the account and all of its Supabase data in one step. **This SQL function must be pasted into the Supabase SQL Editor before the feature works** ŌĆö same manual-apply pattern as the rest of `schema.sql`; there is no migration tooling in this repo that applies it automatically.
- **`ProfileMenu.jsx`** (replaces the earlier `AccountStatus.jsx`) ŌĆö rendered in `App.jsx`'s header (top-right, next to the date/time and the Settings gear icon), visible on every tab. Guest: a plain "Sign in" button (opens `AuthModal`). Signed in: a circular avatar button (the email's first letter) that opens a small dropdown ŌĆö click-outside/Escape-to-close, the same from-scratch pattern as `Select.jsx`. "Sign out" (same `window.confirm` gate as before) is the only entry today; `menuItems` is a plain array specifically so a future entry (e.g. "Account") is just one more array item, not a restructure. Renders nothing while `loading`, same reasoning as the old `AccountStatus.jsx`.

### Cloud sync (Supabase storage provider + migration)

Signing in swaps `storage.js`'s persistence layer from localStorage to Supabase, transparently to every hook ŌĆö no hook file changes. `storage.js` holds `activeProvider` (mutable, defaults to `localStorageProvider`); both it and the new `src/lib/remoteProvider.js` implement the same `get(collection, fallback)` / `set(collection, value)` / `remove(collection)` shape keyed by the existing localStorage key strings (e.g. `'pomodoro_inventory'`), so every `loadX()`/`saveX()` in `storage.js` is unchanged.

- **`remoteProvider.js`** keeps an in-memory `cache` object, always read synchronously by `get()` ŌĆö this is what lets hooks' `useState(() => loadX())` / `useEffect(() => saveX(items), [items])` patterns keep working even though the actual persistence is an async Supabase call. `set()` updates the cache synchronously and fires the upsert in the background (never awaited by the caller). Array collections (`pomodoro_inventory`, `pomodoro_today_tasks`, `pomodoro_activity_log`, `pomodoro_ticks`, `pomodoro_timetable`, `pomodoro_categories`, `pomodoro_void_log`) map to one Supabase table each; `pomodoro_timer_state`/`pomodoro_settings` are one-row-per-user singletons (`onConflict: 'user_id'`). Since `saveX(items)` always passes the *full* authoritative array, `set()` diffs the new array's ids against a `knownIds` cache to compute which ids were removed and issues an explicit `.delete().in('id', ...)` after the upsert ŌĆö an upsert alone never deletes rows.
- **`src/lib/keyCase.js`** ŌĆö converts camelCase JS keys to snake_case Postgres columns and back, with explicit overrides for `diffI`/`diffII` (the generic algorithm can't round-trip these: `diffII` ŌåÆ `diff_i_i` generically, but the schema column is `diff_ii`).
- **One-time migration on sign-in** (`storage.js`'s `signInToRemote(userId)`, called by `App.jsx`): snapshots every current localStorage collection, then `remoteProvider.initializeRemoteData()` fetches each Supabase table and merges local into remote with `mergeCollectionById` ŌĆö the same merge function the JSON-import feature already uses (newer `updatedAt` wins; ties/no-timestamp favor the remote/existing side). Singletons use a simpler rule: an existing remote row always wins outright (never clobbered by a fresh guest session's defaults); a local value is only pushed up if no remote row exists yet. `storage.js` only clears localStorage (`clearLocalGuestData()`) *after* `initializeRemoteData` returns successfully ŌĆö a network/RLS error leaves localStorage completely untouched and surfaces an error banner instead, so a failed sync can never lose data.
- **`App.jsx`** is the auth-transition gate: a `dataMode` state machine (`'guest' | 'syncing' | 'remote'`). Guest mode has zero added latency ŌĆö the render tree mounts immediately, before `useAuth`'s initial session check even resolves. Only an actual sign-in transition (detected via a `prevUserIdRef` comparison, so effect re-fires from reference churn don't trigger duplicate work) shows a brief `<SyncingScreen/>` while `signInToRemote()` runs, then remounts the render tree via a `key` prop change so every hook's `useState` initializer re-runs against the now-warm cache. A dismissible banner shows the "synced to your account Ō£ō" notice once, and a `syncError` banner on failure; both are reset on every new sign-in/sign-out transition so a stale notice can't reappear on a later session.
  - **`appKey` is its own state, set only from inside the auth effect ŌĆö never derived from `user`/`dataMode` at render time.** An earlier version computed it inline as `dataMode === 'remote' && user ? user.id : 'guest'`, which caused a real bug: signing out flips `user` to `null` (from `useAuth`, a separate context) *before* this component's own effect has run, so that inline expression evaluated to `'guest'` ŌĆö and therefore remounted `AppInner` ŌĆö on the very render where `user` went null, which is strictly *before* `signOutFromRemote()` (only called inside the effect, which fires after render/commit) had switched `activeProvider` back to `localStorageProvider` or cleared `remoteProvider`'s cache. The freshly mounted `AppInner`'s hooks (`useInventory` etc.) read storage during that same premature remount, so the "guest" screen kept showing the just-signed-out account's Supabase data. The fix: `setAppKey('guest')`/`setAppKey(userId)` are called explicitly, only *after* `signOutFromRemote()`/a successful `signInToRemote()` have already run synchronously earlier in the same effect ŌĆö so the remount (and the fresh hooks' storage reads) can never race ahead of the provider switch. Regression-tested in `src/App.test.jsx` (mocks `useAuth` and `remoteProvider` only, so it needs no live Supabase project; confirmed to fail against the old inline-derived `appKey` and pass against the fix).
- **Known limitation, not fully resolved**: `remoteProvider.js`'s singleton fetch has one retry after a 300ms delay (`fetchSingletonTableWithRetry`), defending against a hypothetical transient "no row found" result right after sign-in (e.g. two concurrent sign-ins to the same account). This was added while chasing a since-understood-to-be-unrelated bug (a stale notice-banner state, now fixed) and its necessity against a real race was never conclusively proven ŌĆö it's cheap insurance, not a confirmed fix for a confirmed bug.

### Gotchas

- `vite-plugin-pwa` is wired into `vite.config.js` (`generateSW` strategy, `registerType: 'autoUpdate'`) ŌĆö production builds emit a manifest and service worker, making the app installable and offline-capable. It only activates on `npm run build` + `npm run preview` (or a real deploy); `npm run dev` does not register a service worker unless `devOptions.enabled` is set.
- In-app comments and some identifiers are in Turkish; code style otherwise follows standard React/JS conventions.
- Superseded by the i18n system below: user-facing app text is no longer English-only ŌĆö it's translated per the active language (English/Turkish). What still holds is the *reason* behind the old rule: no language should ever leak into the wrong mode (an English string showing up under Turkish, or vice versa) ŌĆö every user-facing string must go through `t()`/the translation dictionaries, never be hardcoded in a component. In-app code comments stay Turkish/English-mixed as before, unaffected by this.
- Deployed to GitHub Pages via `.github/workflows/deploy.yml` on every push to `main`. The workflow runs `npm run lint` and `npm test` before `npm run build`, so a failing test or lint error blocks the deploy. `vite.config.js`'s `base` is only rewritten to `/pomodoro-app/` when the `GITHUB_PAGES` env var is set (the workflow sets it before `npm run build`), so local `dev`/`build`/`preview` stay unprefixed. The build step also forwards `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` from repository secrets ŌĆö Vite only bakes `VITE_`-prefixed env vars present in the process environment at build time into `import.meta.env`, regardless of whether they came from a `.env` file (local) or CI's `env:` block (deploy); without these, the deployed build falls back to `supabaseClient.js`'s `null`-client guest-only path (see Authentication above).
